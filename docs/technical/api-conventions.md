# API Conventions

## URL Structure

All API endpoints live under `/api/`. Resource URLs follow this pattern:

```
/api/<resource>/                     # list / create
/api/<resource>/<id>/                # detail
/api/<resource>/<id>/<action>/       # action (update, deactivate, etc.)
```

Example from the `users` app (`src/users/urls.py`):

```
/api/users/                  GET     UserListApi
/api/users/create/           POST    UserCreateApi
/api/users/<id>/             GET     UserDetailApi
/api/users/<id>/update/      PATCH   UserUpdateApi
/api/users/<id>/deactivate/  POST    UserDeactivateApi
```

URLs are wired in two places:

1. **App-level** — `src/users/urls.py` defines paths relative to the resource.
2. **API-level** — `src/api/urls.py` includes each app under its prefix.
3. **Root-level** — `src/main/urls.py` mounts `api/urls.py` at `/api/`.

## Authentication

The `ApiAuthMixin` (`src/api/mixins.py`) provides Session + Token authentication and requires the user to be authenticated:

```python
from api.mixins import ApiAuthMixin

class MyApi(ApiAuthMixin, APIView):
    ...
```

Token authentication uses the `Authorization` header:

```
Authorization: Token <token-value>
```

Tokens are managed through DRF's built-in `rest_framework.authtoken` app.

## Error Format

Every error response uses a consistent shape, enforced by the custom exception handler:

```json
{
    "message": "Human-readable error description.",
    "extra": {
        "fields": {
            "email": ["This field is required."]
        }
    }
}
```

- `message` — always a string, suitable for displaying to users.
- `extra` — object with additional context. `fields` is present for validation errors.

See [Architecture — Error Flow](architecture.md#error-flow) for implementation details.

## Pagination

The default pagination is `LimitOffsetPagination` (limit=50, max=100).

| Class | Default Size | Max Size | Query Params |
|---|---|---|---|
| `LimitOffsetPagination` | 50 | 100 | `?limit=N&offset=N` |
| `SmallPagePagination` | 10 | 50 | `?page=N&page_size=N` |
| `MediumPagePagination` | 25 | 100 | `?page=N&page_size=N` |
| `LargePagePagination` | 50 | 200 | `?page=N&page_size=N` |

Use `get_paginated_response()` from `src/api/utils.py` to paginate in views:

```python
from api.pagination import LimitOffsetPagination
from api.utils import get_paginated_response

def get(self, request):
    users = user_list(filters=filter_serializer.validated_data)
    return get_paginated_response(
        pagination_class=LimitOffsetPagination,
        serializer_class=self.OutputSerializer,
        queryset=users,
        request=request,
        view=self,
    )
```

## Filtering

Filtering uses `django-filter` with the **FilterSerializer pattern**:

1. Define a `FilterSet` in `filters.py` for the model.
2. Add a nested `FilterSerializer` in the API view to validate query params.
3. Pass the validated data to the selector.

```python
# In the API view
class FilterSerializer(serializers.Serializer):
    email = serializers.CharField(required=False)
    is_active = serializers.BooleanField(required=False, allow_null=True, default=None)

def get(self, request):
    filter_serializer = self.FilterSerializer(data=request.query_params)
    filter_serializer.is_valid(raise_exception=True)
    users = user_list(filters=filter_serializer.validated_data)
    ...
```

## Field Selection and Expansion

`DynamicModelSerializer` (`src/utils/serializers.py`) supports two query parameters:

- **`?fields=id,email,name`** — return only the specified fields.
- **`?expand=profile,company`** — include nested serializer fields.

Configure expandable fields in the serializer's `Meta`:

```python
class UserSerializer(DynamicModelSerializer):
    class Meta:
        model = User
        fields = "__all__"
        expandable_fields = {
            "profile": (ProfileSerializer, {}),
            "company": (CompanySerializer, {"source": "company"}),
        }
```

## Inline Serializers

For one-off response shapes, use `inline_serializer()` instead of defining a full class:

```python
from api.utils import inline_serializer

serializer = inline_serializer(fields={
    "email": serializers.EmailField(),
    "is_active": serializers.BooleanField(),
})
```

## OpenAPI / Swagger

The API schema is generated by [drf-spectacular](https://github.com/tfranzel/drf-spectacular).

| Endpoint | Description |
|---|---|
| `/api/schema/` | Raw OpenAPI 3.0 schema (YAML) |
| `/api/schema/swagger/` | Swagger UI |
| `/api/schema/redoc/` | ReDoc |

Configuration lives in `src/main/settings/spectacular.py`.
